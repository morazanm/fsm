procedure

(make-dfa sts    
  sigma    
  start    
  finals    
  delta    
  add-dead    
  [#:rejects rejects    
  #:accepts accepts]) → dfa
  sts : (listof state)
  sigma : alphabet
  start : state
  finals : (listof state)
  delta : (listof dfa-rule)
  add-dead : (or/c 'no-dead void?)
  rejects : (listof word) = '()
  accepts : (listof word) = '()
Builds a deterministic finite-state automaton. delta is a transition function. The optional argument add-dead is used to signal the constructor to add or not add a dead state and transitions into it for any transitions in delta (to make it a function). If the delta function is fully specified providing ’no-dead prevents the addition of such a dead state and associated transitions. The optional keyword parameters rejects and accepts are used to integrate testing. They are both lists of words using elements of sigma. The first is a list of words that ought to be rejected and the second is a list of words that ought to be accepted. If any word produces the wrong result the constructor fails and a recipe-based error is thrown.

  delta : (listof pda-rule)
  rejects : (listof word) = '()
  accepts : (listof word) = '()
Builds a nondeterministic pushdown automaton from the given list of states, alphabet, list of stack symbols, start state, list of final states, and list of pda-rule. delta is a transition relation. The optional keyword parameters rejects and accepts are used to integrate testing. They are both lists of words using elements of sigma. The first is a list of words that ought to be rejected and the second is a list of words that ought to be accepted. If any word produces the wrong result the constructor fails and a recipe-based error is thrown.

Given that a nondeterministic pda may only semidecide a language, use rejects with caution. Words not in the machine’s language may cause the constructor to not terminate.

Builds a nondeterministic Turing machine. delta is a transition relation. LM is automatically added to the machine’s alphabet. Rules for moving off the LM are automatically added to the machine’s rules. If the optional accept argument is given then the resulting Turing machine is a language recognizer. The optional keyword parameters rejects and accepts are used to integrate testing. They are both lists of words using elements of sigma. The first is a list of words that ought to be rejected and the second is a list of words that ought to be accepted. If any word produces the wrong result the constructor fails and a recipe-based error is thrown.

Given that a tm may only semidecide a language, use rejects with caution. Words not in the machine’s language may cause the constructor to not terminate.

Builds a state machine that is excatly the same as the given machine except that its states are renamed as to not have a name in common with the given list of states.

procedure

(sm-union m1 m2) → machine?

  m1 : machine?
  m2 : machine?
Builds a state machine for the language obtained from the union of the languages of the two given state machines. If the inputs are Turing machines then they must be language recognizers. The given machines must have the same type. May not be used with multitape Turing machines.

procedure

(sm-concat m1 m2) → machine?

  m1 : machine?
  m2 : machine?
Builds a state machine for the language obtained from the concatenation of the languages of the two given state machines. [Note: Not yet implemented for Turing machines language recognizers. May not be used with multitape Turing machines.].

procedure

(sm-kleenestar m1) → machine?

  m1 : machine?
Builds a state machine for the language obtained from the Kleene star of the given machine’s language. May not be used with multitape Turing machines nor with Turing machine language recognizers.

procedure

(sm-complement m1) → machine?

  m1 : machine?
Builds a state machine for the language obtained from the complement of the given machine’s language. The given machine can not be a ndpda. If the inputs are Turing machines then they must be language recognizers. May not be used with multitape Turing machines.

procedure

(sm-intersection m1 m2) → machine?

  m1 : machine?
  m2 : machine?
Builds a state machine for the language obtained from the intersection of the languages of the two given state machines. If the inputs are Turing machines then they must be language recognizers. The given machines must have the same type. May not be used with multitape Turing machines.

procedure

(grammar->sm g) → machine?

  g : grammar
Builds a state machine for the language of the given regular or context-free grammar.

procedure

(fsa-minimize m) → dfa

  m : fsa
Builds a deterministic finite-state automaton with a minimal number of states from the given finite-state automaton.
